Containerized Web App on AWS

Deploying a Static Web App with Docker, ECR, ECS, and ALB

This project takes a simple HTML site and treats it like a real cloud workload. The app is containerized with Docker, stored in Amazon ECR, and deployed to AWS Fargate (ECS) behind an Application Load Balancer.

AWS Services Used

Each service has a clear job. Together they form a simple but realistic container deployment pipeline.

Step 1

Amazon EC2 + Docker

I launched an Amazon Linux EC2 instance and installed Docker. On this machine I created a Dockerfile that installs Apache HTTPD and serves this index.html page, then built an image called project-image.

Step 2

Amazon ECR (Image Registry)

I created a private Amazon ECR repository, logged in with the AWS CLI using an IAM access key, tagged my local Docker image with the ECR URI, and pushed it to the registry as the source of truth for deployments.

Step 3

ECS Task Definition + Fargate

I defined an ECS task definition that pulls the ECR image, runs the Apache container, and exposes port 80. The compute type is AWS Fargate, which means the containers run on serverless infrastructure instead of manual EC2 instances.

Step 4

Application Load Balancer

I created an Application Load Balancer that listens on HTTP port 80 and forwards requests to a target group mapped to the ECS service. With multiple tasks running, the ALB can balance traffic across containers and keep the app available if one task fails.

Deployment Timeline

High level view of the steps I followed from local HTML file to a fully managed container service on AWS.

Phase 1
Local project setup
Wrote a static index.html landing page and created a Dockerfile that installs Apache HTTPD on Rocky Linux and serves the site on port 80.
Phase 2
Image build on EC2
Connected to the EC2 instance with SSH, installed Docker, and ran docker build -t project-image . to create the container image for the app.
Phase 3
Push to Amazon ECR
Created an ECR repository, logged in using aws ecr get-login-password, tagged the image with the ECR URI, and pushed it so ECS can pull it on demand.
Phase 4
ECS service with Fargate tasks
Created an ECS cluster and service using Fargate, based on the task definition. Set the desired count to 2 tasks so the application runs in multiple Availability Zones.
Phase 5
Expose through Application Load Balancer
Attached the service to an Application Load Balancer listener on port 80, so users hit a single DNS name and the ALB routes requests to the healthy tasks.

Architecture at a Glance

Simple, container focused, and production inspired.

  • Static HTML page served by Apache HTTPD inside a Docker container.
  • Container image stored in a private Amazon ECR repository.
  • ECS Fargate runs 2 copies of the container across multiple subnets.
  • An Application Load Balancer listens on port 80 and forwards traffic to the ECS tasks through a target group with health checks.
  • All traffic flows through security groups that control HTTP and SSH access.
Built as a learning project to practice containerization, AWS ECR, ECS Fargate, and ALB.
